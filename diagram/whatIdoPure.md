个人工作总结

一、 主要工作内容

1. 前端架构搭建与开发
   - 整体架构把控：负责项目前端部分的整体架构搭建。选用 Nuxt.js 作为核心开发框架，利用其约定式路由（File-system Routing）的特性，无需繁琐的路由配置，极大加快了项目的构建过程与开发效率。
   - 界面布局与设计：引入 Tailwind CSS 进行样式开发。基于其原子化 CSS 的理念，实现了界面的快速布局与样式复用，简化了设计流程，确保了设计风格的一致性与现代码代码的整洁度。
   - 核心功能实现：负责了项目中绝大多数核心页面的编写，主导了前端业务逻辑的实现。

2. 前端性能深度优化
   - 自研懒加载 Hook：针对项目日益复杂导致的启动速度问题，自主设计并编写了 Lazy Load Hook。
   - 按需加载策略：利用该 Hook 对评论区、商品列表等“重”组件进行了重构，实现了按需加载 (On-demand Loading)。仅在用户滚动到可见区域或需要交互时才加载相关资源，显著降低了首屏资源体积，大幅提升了页面的启动速度与渲染性能。

3. AI 智能体后端开发 (Python)
   - AI 后端架构：独立承担了智能体（Agent）的后端开发工作。采用 Python 技术栈，基于 FastAPI 高性能 Web 框架结合 LangChain 编排框架，从零搭建了 AI 服务端。
   - 核心逻辑链路：实现了从“接收用户消息” -> “上下文管理” -> “消息存储” -> “模型推理” -> “结果返回”的完整闭环逻辑。
   - Tools (Function Calling) 开发：为了拓展 AI 的能力边界，开发了多个关键的 Function Tools：
     * 文生图工具：封装了第三方文生图接口（如通义千问等），赋予 AI 根据文本描述生成图片的能力。
     * 智慧商城服务：打通了 AI 与商城业务数据。当用户询问商品信息（如“查一下某商品价格”）时，AI 可自动调用该工具查询数据库并返回实时商品信息，实现了业务数据与大模型的深度融合。

二、 遇到问题与解决方案

1. 复杂页面加载性能瓶颈
   - 问题描述：随着商城业务功能的丰富，部分页面（如首页、商品详情页）集成了大量图片与交互组件，导致应用启动速度变慢，首屏渲染时间（FCP）过长，影响用户体验。
   - 解决方案：
     * 深入分析前端渲染机制，决定采用“时间换空间”的策略。
     * 自主开发 Lazy Load Hook：利用 IntersectionObserver API 监听元素可见性。
     * 实施效果：将非首屏核心组件（如底部的评论列表、推荐商品）通过该 Hook 进行封装，实现了真正的“懒加载”。经测试，页面初始化时的网络请求数量显著减少，启动速度得到明显提升。

2. 大模型与业务数据“隔离”问题
   - 问题描述：通用大语言模型缺乏项目内部的实时业务数据（如商城库存、具体商品详情），导致用户咨询具体业务问题时，AI 经常产生“幻觉”或回答无法提供服务。
   - 解决方案：
     * 基于 LangChain 框架，引入 Function Calling (工具调用) 机制。
     * 开发业务工具：编写 Python 函数直接对接商城后端数据库 API。
     * Prompt 调优：在 System Prompt 中详细定义工具的描述与参数格式，引导模型在识别到“查询商品”意图时准确调用工具。
     * 通过这一方案，成功将“静态”的大模型与“动态”的业务数据库连接，解决了信息隔离问题。

三、 个人总结与体会

通过本次智慧社区项目的开发，我完成了从传统的 Web 前端开发 向 全栈及 AI 应用开发 的角色跨越。

在前端领域，我深刻体会到了 Nuxt + Tailwind CSS 组合在构建现代化 Web 应用时的开发体验优势，同时也意识到性能优化不能仅停留在工具配置层面，更需要针对具体业务场景（如长列表、重组件）进行底层的逻辑优化（如自研 Hook）。

在后端与 AI 领域，这是我首次深度实践 Python + LangChain 的开发模式。通过亲手搭建 AI Backend 并实现 Function Calling，我理解了“大模型即操作系统，Tools 即 APP”的理念。让 AI 能够“并在”业务系统中真正“干活”（查数据、画图），而不仅仅是陪聊，是我在本项目中获得的最大成就感来源。这次经历极大地拓宽了我的技术视野，也为我未来在 AI Native 应用开发方向打下了坚实基础。
